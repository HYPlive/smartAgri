ABOUT PROJECT

解决农田监测中“看得见、查得到、能派单、可执行”的问题

传统巡田依赖人工经验，地块分布、遥感影像、异常结果和巡检任务之间常常割裂。这个项目将地块作为核心数据对象，把地图圈地、遥感处理、异常检测、工单调度、路径规划和 AI 助手整合到一个系统里，让异常从“检测结果”变成“可执行任务”。

WHY THIS PROJECT

项目痛点与解决思路

页面里不只展示功能，也说明为什么这样设计，以及系统如何把农业场景中的碎片化数据组织成闭环。

PAIN 01

地块数据分散

地块边界、作物、土壤、灌溉和地址信息如果只靠表格维护，难以和地图空间位置形成联动。

PAIN 02

遥感影像体积大

整幅 GeoTIFF 传输和渲染成本高，需要按地块边界裁剪后再按需展示。

PAIN 03

异常结果难执行

检测结果如果没有转成工单和路线，只能停留在分析层，无法指导巡田任务。

PAIN 04

查询依赖人工筛选

管理员需要通过自然语言快速查询地块和异常信息，而不是反复组合筛选条件。

AI AGENT WORKFLOW

AI 不是普通聊天，而是能调用业务工具的 Agent

系统将用户问题交给大模型判断意图，再通过 Function Calling 调用业务工具，最后通过 SSE 把执行过程和结果返回前端。

1. 自然语言输入

管理员提出“帮我查沽源的地块”“最近有哪些异常工单”“帮我规划巡田路线”等问题。

2. 意图识别

LLM 根据系统提示判断是否需要查询地块、获取异常记录、触发检测或生成路线。

3. 工具调用

调用 query_plots、get_anomaly_records、trigger_anomaly_detection、generate_patrol_route 等业务工具。

4. 结果组织

将数据库记录、异常工单和路径结果转成自然语言回答，并补充管理建议。

5. 流式展示

前端通过 SSE 展示 AI 回复、工具调用过程和最终结果，提升交互反馈速度。

TECHNICAL HIGHLIGHTS

技术难点与解决方案

这些内容适合放在面试讲解中，能体现项目不是简单后台 CRUD。

GeoTIFF 按地块裁剪

影像数据体积大，直接加载整幅图会影响地图交互。通过 Flask 服务按地块边界裁剪，前端只加载需要查看的区域。

异步巡田任务处理

异常检测属于耗时计算，系统通过异步任务和轮询获取结果，避免主业务请求长时间阻塞。

AI 工具调用编排

将地块查询、异常记录、检测触发、路线规划封装成工具，让 LLM 根据用户问题自主选择调用路径。

地图空间交互

前端需要处理地块绘制、坐标保存、地图定位、影像叠加和三维展示，保证空间信息能直观呈现。

INTERVIEW SUMMARY

这是一个面向精准农业场景的全栈项目，我主要完成了业务接口、地图交互、Python 算法服务集成和 AI Agent 能力接入。项目亮点在于把遥感异常检测结果进一步转化为巡田工单和路径规划，并通过 AI 助手降低管理员查询和决策成本。
